# 数据库设计与微服务模块映射说明

## 1. 说明范围

本文依据 `C:\Users\23883\Desktop\fan-mes-workspace\design\database_design\数据库设计.md` 整理，是对既有数据库设计的阅读、总结与微服务模块映射说明。

本文不重新设计数据库表，不修改表名、字段名、字段类型、主键、外键或业务含义。若数据库设计中的业务域与当前微服务目录结构存在不完全一致，本文仅给出对齐建议。

## 2. 数据库/逻辑库划分

原数据库设计按业务域和表名前缀划分逻辑数据域：

| 逻辑数据域 | 表名前缀 | 主要内容 |
|---|---|---|
| 系统权限与审计 | `sys_` | 用户、角色、菜单、权限、数据范围、状态日志、操作日志、编号规则 |
| 基础主数据 | `md_` | 组织、车间、产线、工位、产品、物料、客户、供应商、计量单位、班组、BOM |
| 生产订单 | `prod_` | 生产工单、齐套分析、欠料明细、派工单 |
| 现场管理 | `shop_` | 生产任务单、工序作业、生产状态、报工记录、完工单、返修工单 |
| 工艺管理 | `route_`、`sop_` | 工序、工艺路线、路线工序、产品路线、电子 SOP、不良原因 |
| 条码追溯 | `bc_`、`trace_` | 条码类型、条码规则、模板、应用规则、条码实例、打印记录、追溯事件 |
| 设备管理 | `eqp_` | 设备类别、制造商、台账、点检、保养、报修、维修、OEE、能源 |
| 设备对接 | `iot_` | 设备采集点位、原始采集记录、设备计数报工、设备联调日志 |
| 质量管理 | `qc_` | 检验项目、检验方案、检验单、检验结果、不良记录 |
| 安灯管理 | `andon_` | 安灯类型、异常配置、异常原因、异常单、通知记录、处理记录 |
| 计件工资 | `wage_` | 工序计件单价、计件工资明细、计件工资汇总 |
| 报表看板 | `rpt_`、`board_` | 指标定义、指标快照、查询条件、看板配置、看板数据缓存 |

## 4. 主要表关系

生产主链路关系：

| 上游表 | 下游表 | 关系说明 |
|---|---|---|
| `prod_work_order` | `prod_dispatch_order` | 一个生产工单可生成多个派工单 |
| `prod_work_order` | `shop_task` | 一个生产工单可形成多个生产任务单 |
| `prod_dispatch_order` | `shop_task` | 派工单下发后形成或关联生产任务单 |
| `shop_task` | `shop_operation_task` | 一个生产任务单包含多个工序作业 |
| `route_header` | `route_step` | 一条工艺路线包含多个路线工序 |
| `route_step` | `shop_operation_task` | 路线工序驱动现场工序作业 |

## 7. 对齐建议

1. 当前微服务目录没有独立的 `mes-route`、`mes-barcode`、`mes-andon`、`mes-wage` 服务。建议在不修改数据库表设计的前提下，暂由 `mes-production` 承接工艺、条码追溯、安灯和计件工资相关表，后续若拆分服务，应仅迁移服务边界，不改表结构和业务含义。
2. `sys_` 表同时覆盖认证、授权、菜单、审计、编号等能力。建议 `mes-auth` 主责身份与权限关系，`mes-system` 主责菜单、状态日志、操作日志和编号规则，二者通过接口或公共模块协作。
3. `iot_` 表属于原设计中的设备对接域，但当前没有独立设备对接服务。建议由 `mes-equipment` 主责设备点位、采集、计数报工和联调日志，`mes-integration` 仅主责外部系统同步审计与幂等日志。
4. `rpt_`、`board_` 表是分析和缓存类数据，建议 `mes-report` 只持有指标定义、指标快照、查询条件和看板缓存，源数据仍由生产、质量、设备、安灯、追溯等业务模块维护。
5. 所有跨模块状态变化仍应按原设计写入 `sys_status_log`；所有外部写入仍应先写 `int_sync_log` 并校验幂等，再执行业务落库。
